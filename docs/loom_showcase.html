系统架构 —— 七源汇流,一份真相,派生一切

像织机:6 股采集来源汇入一个结(归一 + 打码)→ 一份真相 → 织出检索、日记、主题、云备份。 日报不是采集源——它是 AI 读本库其它数据合成的产出,渲染回日记顶部(见右侧)。

核心数据模型 —— 所有来源归一成同一结构

一条 entry = "一件发生过的事 / 一份材料"。不同来源塞不同的 detail 字段,不为每种源建表。

{
  "id":      "git:data-marketing:0842ee76",   // 稳定主键 → 幂等去重 / 可按天拆
  "date":    "2026-07-02",  "ts": "…T22:30:00",  "project": "data-marketing",
  "tool":    "git",   "kind": "commit",   // 来源 / 类型
  "summary": "fix(dwb-moloco): clean_name 补剥去重尾…",
  "ref":     "0842ee76",       // 回链原文,不复制全文
  "detail":  { // 因源而异:body / opening / file_list / content / path / dated … }
}
id 稳定 重复采集只覆盖;可编码天(claude:会话:日期)。
ref 回链 只存指针+摘要,全文留原处,库保持轻。
detail 自由 每种源塞各自字段,无需多表。
扁平存储 不按天/项目分桶——那是"视图"。

可检索的字段 —— 先认识这几个词

下面每类数据卡上的「检索」标签,写的都是这几个字段名。它们是一条 entry 的 detail 里 因源而异的文本槽:标 ◆ 的进 FTS 索引,能被 loom search 搜到; ref 只作回链、不进全文。(卡片上的字段名可鼠标悬停看释义)

summary
一句话标题 / 摘要,每条必有。 git = 提交标题 · AI 会话 = 意图标题 · 数据 = 卡名 · 文档 = H1 ◆ 进检索 · 也是日记/搜索结果里最先看到的那行
opening
会话的「开场首问」全文(仅 AI 会话类)。你打开这个 session 说的第一句真实提问。 ◆ 进检索
body
长正文。 git = 提交正文(为什么改 + 怎么验证)· Claude = 当天全部提问合并(≤8000 字) ◆ 进检索 —— 所以整段提问 / 提交理由都搜得到
headings
文档大纲:各级标题拼成的目录(仅文档类)。用它定位"这文档讲了啥"而不必读全文。 ◆ 进检索
content
文档 / 代码全文快照(≤10 万字,超长尾部不进)。源文件删了、这份快照还在。 ◆ 进检索
digest
AI 会话摘要:读当天会话的「问 + 答」原文后,AI 写的准确标题 + 2–4 句摘要(可选生成)。 补"首问含糊 / 答案侧搜不到"的洞;覆盖后日记里该会话标题带 ✦。 ◆ 进检索(存独立 sidecar,重采不丢)
ref
回链指针:git hash / jsonl 路径 / 文件名。点它一跳回到原文看完整 diff / 完整对话。 ○ 不进检索,只作溯源(库保持轻)

为什么这样拆? 一条记录只留"值钱的短文本 + 一个回链",全文/整 diff/完整对话都留在原处按需回溯—— 这就是库能塞下 1200+ 条却依然轻、且每条都可一跳还原的原因。detail 是装这些槽的"字段袋", 不为每种源单独建表。

每种数据:怎么采 · 怎么处理 · 存成啥 · 取舍

七类数据各有各的解析与存储策略。统一原则:只存摘要+回链+可检索文本,全文/原始留在原处; 能上云的是文本知识层,重/敏感的留本地。每张卡底部可展开「看这类长啥样」——是本库里的真实条目。

诚实的取舍

明知的边界,写出来才可信。

  • 检索是关键词,不是语义 — "cohort" 召不回"留存";语义检索要向量模型,会破坏零依赖,暂不上。
  • 只 Claude 能逐条拆会话 — Cursor/Codex 无消息级文本,只能两端归属。
  • 助手回答默认不进检索 — 采集只索引你的提问;需要时用 loom session gen 让 AI 读当天问+答生成摘要,把答案侧内容也纳入检索(存独立 sidecar,重采不丢)。
  • 全量重建 — 每次 sync 全量重扫;单人无感,超大历史才需增量。
  • AI 分类非全自动 — 闭集 + 人过目 + 审计控准。
