# loom-bot 设计文档 —— 飞书记事机器人(多租户)

> 状态:蓝图(未实现)。等具备 ①一台常驻机器 ②AI API(可选,热插拔)后开工。
> 与本地 `loom` 的关系见下。

## 1. 第一性目标

给每个人一个**零摩擦的结构化记事入口**:在飞书任何话题里 @ 机器人一句,
它就读上下文、替这个人记进**仅他可读的多维表格**,并回执确认。

很多真实需求/结论走群聊而非需求池;人也懒得手动登记。loom-bot 把
「说一句 @机器人」变成「自动归档到我的私有库」,消除登记成本。

## 2. 与本地 loom 的关系(姊妹组件)

| | loom(已建) | loom-bot(本文) |
|---|---|---|
| 视角 | 只有我 | 任何人 |
| 运行时 | 本地 CLI / cron 拉取 | 托管、常驻、事件驱动 |
| 数据落地 | 我机器上的 markdown vault | 每个用户的飞书多维表格 |
| 触发 | 定时 sync | 实时 @机器人 |

两者哲学一致(**结构化捕获**),以**多维表格为契约**解耦:loom-bot 负责实时捕获+AI 总结+
写表(带原消息回链),loom 只负责拉取式读表(现成的 `feishu` collector)。
个人自用一张表即可;全员形态则每人一张仅本人可读的表。
(早期 loom 内曾有 `feishu_im` 采集器直接读 IM,已退役——读 IM 的职责归 bot。)

## 3. 运行时

- **常驻长连接服务**,订阅事件 `im.message.receive_v1`(飞书推荐长连接,免公网 Webhook)。
- 宽读权限已确认可开(`im:message:readonly`),故可读到被 @ 消息的话题上下文。
- 无状态化处理 + 一张主 bitable 存「用户→base」注册表(下节),服务重启不丢映射。

## 4. 所需权限(scope)

| scope | 用途 |
|---|---|
| `im:message:readonly` | 读被 @ 消息及其话题上下文 |
| `im:message:send_as_bot` | 回执「✅ 已记录」 |
| `bitable:app` | 建/写多维表格 |
| `drive:drive` | 设置文档协作者 / 公开权限 |
| `contact:user.base:readonly` | 解析 open_id → 姓名 |

## 5. 端到端时序

```
用户在话题 @机器人:"记一下 BF 分月需求,已完成"
        │
        ▼ im.message.receive_v1 事件(sender open_id, message_id, thread_id/root_id, 文本)
1. 读上下文    GET im/v1/messages(按 thread/root 取话题若干条)
2. 认领 base   查主注册表 registry: open_id -> {app_token, table_id}
                 命中 → 复用;未命中 → 走 3
3. 建私有 base POST bitable/v1/apps  {name:"<姓名> 的记事", folder_token}
                 POST .../tables      建表(字段见 §7)
                 PATCH drive 权限:加该用户为 full_access、关公开链接
                 写回注册表
4. 写记录      POST bitable/v1/apps/{app_token}/tables/{table_id}/records
                 (有 AI:结构化字段;无 AI:原文+元数据)
5. 回执        POST im/v1/messages reply  "✅ 已记录 → <base 链接>"
```

## 6. 权限模型(据实说明)

严格「**仅他可读**」与「**他 + app 可读写**」有取舍:app 必须保留**写权限**才能
持续帮他记。可行方案:

- 建 base 时 owner 为 app(tenant);**加目标用户为 full_access 协作者**;
  **关闭公开/组织可见**(`drive/v1/permissions/{token}/public` 设为不可访问)。
- 结果 = 「该用户 + app 可读写,其他人不可见」,而非字面「只有他一个」。
- 可选进阶:把 owner **转让给用户**(`drive` transfer_owner),app 降为 editor。
  更贴近「他的表」,但转让后若 app 被移除协作者则无法再写,需权衡。

## 7. 数据模型

**主注册表(一张 bitable)**:`open_id | 姓名 | base_app_token | table_id | 创建时间`。

**每用户记事表字段**:
`标题 | 原文 | 状态(待办/进行/已完成) | 分类 | 截止 | 来源群 | 话题链接 | 记录时间`
无 AI 时:标题=原文首句、状态默认「待办」、其余留空。

## 8. AI 接入点(热插拔增强,现在缺 API 不阻塞)

- 输入:@ 消息文本 + 话题上下文。
- 输出:结构化字段(标题 / 状态 / 截止 / 分类),以及「这是新需求还是对旧条目的状态更新」。
- 无 AI:存原文 + 元数据,表照常可用;AI 到位后仅升级抽取质量与去重/状态更新能力。
- 状态更新:同一话题后续 @「done」→(AI 或规则)将该话题已有记录置为「已完成」。

## 9. 多租户即隐私

替他人读话题、建库 = 在替别人存数据。**必须**:

- 走**管理员授权** + 用户**知情同意**(首次 @ 时回执里声明并给「停用」入口)。
- 每人数据物理隔离在各自 base;服务日志不落原文正文。
- 提供退出:用户可要求删除其 base 与注册项。

## 10. 分阶段路线

1. **MVP(无 AI,单群)**:@机器人 → 建/复用我的 base → 存原文+元数据 → 回执。验证核心链路。
2. **结构化(接 AI)**:自由文本 → 字段;支持「done」状态更新与去重。
3. **全员**:注册表规模化、管理员授权流程、退出/删除、审计。

## 11. 待拍板

- base 归属:app 持有 + 用户协作 vs 转让用户。
