Architecture — Multiple streams, one truth, derive everything

Like a loom: multiple ingestion sources converge at a knot (normalize + redact) → one source of truth → weave out search, journal, topics, cloud backup. Daily reports are not an ingestion source — they are AI-synthesized output derived from other data already in the store, rendered at the top of each day's journal (see right).

Core data model — All sources normalized to the same shape

One entry = "one thing that happened / one piece of material." Different sources fill different detail fields — no separate table per source type.

{
  "id":      "git:data-marketing:0842ee76",   // stable primary key → idempotent dedup / day-bucketing
  "date":    "2026-07-02",  "ts": "…T22:30:00",  "project": "data-marketing",
  "tool":    "git",   "kind": "commit",   // source / type
  "summary": "fix(dwb-moloco): clean_name strip dedup suffix…",
  "ref":     "0842ee76",       // back-link to original; full text stays at source
  "detail":  { // source-specific: body / opening / file_list / content / path / dated … }
}
Stable id Re-ingestion only upserts; id can encode date (claude:session:date).
ref back-link Stores pointer + summary; full text stays at source, keeping the store lean.
Free-form detail Each source fills its own fields — no extra tables needed.
Flat storage Not bucketed by day/project — those are "views."

Searchable fields — Know these terms first

The "Indexed" tags on each type card below refer to these field names. They are the source-specific text slots inside an entry's detail: Fields marked ◆ are indexed in FTS and reachable by loom search; ref is a back-link only — not full-text indexed. (Hover over the field chips on each card for a tooltip.)

summary
One-line title / summary — every entry has one. git = commit subject · AI session = intent title · data = card name · doc = H1 ◆ Indexed · the first line shown in journals and search results
opening
Full text of the session's opening message (AI session types only). The literal first real question you typed in that session. ◆ Indexed
body
Long-form text. git = commit body (why + how verified) · message-capable AI tools = all your prompts for that day merged (≤8,000 chars) ◆ Indexed — full prompts and commit rationale are all searchable
headings
Document outline: all heading levels concatenated (doc type only). Lets you locate "what this doc covers" without reading the full text. ◆ Indexed
content
Full-text snapshot of a doc or code file (≤100k chars; tail truncated if longer). Snapshot survives even if the source file is deleted. ◆ Indexed
digest
AI session digest: after reading the day's full Q&A, the AI writes an accurate title + 2–4 sentence summary (optional generation). Fills the gap when the opening prompt is vague or the answer-side content is unsearchable; once written, the journal title gains ✦. ◆ Indexed (stored in a sidecar file; re-ingestion never clobbers it)
ref
Back-link pointer: git hash / jsonl path / filename. Follow it to see the full diff or complete conversation. ○ Not indexed — traceability only (keeps the store lean)

Why this split? Each entry keeps only "valuable short text + one back-link"; full diffs, transcripts, and raw files stay at the source for on-demand retrieval. That is why the store holds 1,200+ entries and stays lean, while every entry is one hop away from its original. detail is the field bag for these slots — no separate table per source type.

Per-type breakdown: how each data type is collected, processed, stored, and why

Seven data types, each with its own parsing and storage strategy. Shared principle: store summary + back-link + searchable text only; full text and raw files stay at source. What can go to the cloud is the text-knowledge layer; heavy or sensitive files stay local. Expand "What does it look like?" at the bottom of each card to see a real entry from the store.

💻git commitstool=gitkind=commit
Source
Each repo via git log --all --no-merges --since, filtered to your own commits by email/name
Processing
Parse header + body (%b) + file stats (numstat); sentinel-separate body from numstat; renames use new path; dedup on same subject + same delta
Stored as
summary = subject; detail = {body ≤4,000, file_list ≤40 (with +/- lines), ins/del/files}; date = author date
Trade-off Stores the subject + "why it changed" body + file stats, but not the full diff (ref = hash; use `git show` for the full diff) — keeps the store lean while preserving the most valuable decision context.

🧮Data files csv / xlsxloom data addkind=note
Source
Manual loom data add (query exports, pipeline outputs)
Processing
Pure stdlib parsing (csv module / xlsx via zip+xml, handling inlineStr, shared strings, date serial number restoration) → distills a "data card"
Stored as
Data card .card.md (columns/types/stats/top-5 sample rows + lineage inputs/produced_by/used_by) into vault; raw file copied to _data/

Source
Code files ingested into vault/notes via loom doc add
Processing
Ingestion-time redaction (values only, variable names preserved); date priority: frontmatter > filename date > mtime
Stored as
Code stored verbatim (searchable); files without a reliable date are flagged dated=False

Honest trade-offs

Known limitations, stated plainly.

  • Search is keyword-based, not semantic — "cohort" won't recall "retention"; semantic search needs embedding models, which would break the zero-dependency constraint. Not planned for now.
  • Cursor still uses two-endpoint attribution — Claude/Codex/CodeBuddy/pi/OpenCode split by message date; Cursor exposes session-level timestamps only.
  • Assistant replies are not indexed by default — the collector indexes only your prompts; use loom session gen to have the AI read the full Q&A and generate a digest that brings answer-side content into the index (stored in a sidecar; re-ingestion safe).
  • Full rescan on every sync — each sync rescans everything; imperceptible for a single user, incremental mode only needed for very large histories.
  • AI classification is not fully automatic — closed-set labels + human review + audit for precision.
