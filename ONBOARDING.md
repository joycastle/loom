# ONBOARDING —— 给 AI 助手的执行剧本

> 读者是 **AI 助手(Cursor / Claude Code / Codex)**,不是人。用户 clone 了 loom,想把自己
> 散落各处的工作痕迹(git 提交 / AI 对话 / 文档 / 代码 / 数据 / 日报)整理成一本可检索、
> 可按主题追溯的台账。**你的任务:照这份剧本,一步步带 ta 完成配置 + 整理历史。**

## AI 须知(先读,全程遵守)
- 这是**执行剧本**:按阶段做,做完一步再下一步;每步先看命令输出再决定下一步。
- **不可逆 / 外发操作先向用户确认**:删除、移动文件、`--push`(推云)、建远端仓、写 cron。
- **凭证只进 `~/.loom/.env`(chmod 600),绝不写进任何仓、任何 md、任何输出**。
- **分类要看内容,别图快**:这是本项目**最容易翻车**的地方。靠关键词批量分类必然过采/误挂;
  务必逐条读内容判断、再对抗复核(见阶段 E)。宁缺勿滥,拿不准就不分。
- 命令 `loom` 若不存在,用 `python3 -m loom`(在仓库根目录)。

## 阶段 A — 环境与实例
**目标:装好、能跑、建实例。**

1. 可运行性检查:
   ```
   loom --help  ||  python3 -m loom --help
   python3 -m pytest tests/ -q        # 应全绿,确认代码完好
   ```
2. **问用户**:你的名字?git 提交邮箱(可多个)?要扫描哪个目录找 git 仓(默认 `~/Documents`)?
3. 跑引导(交互式,让用户自己填,或你据上一步代填):
   ```
   loom init
   ```
   > **为什么**:`init` 写 `~/.loom/config.json`,并扫描纳入 git 仓。
4. 讲清两处分离(建立正确心智):
   - **代码仓** = 这个项目本身(工具逻辑),可公开/分享。
   - **实例** = `~/.loom/`:`config.json`(配置)、`.env`(凭证)、`data/`(真相 entries.jsonl + 派生索引 + 主题映射)、`vault/`(markdown 输出:日记 + notes)。**实例是私有的,永不进代码仓。**

## 阶段 B — 首次采集
**目标:把已有痕迹拉进台账。**

```
loom sync          # 采集所有源 → 归一 → 打码 → 建索引 → 渲染日记(不推云)
loom today         # 看今天
loom search 关键词  # 验证可检索(中文子串 OK)
```
讲给用户(简短):
- **来源**:git 提交、Claude/Cursor/Codex 会话、各仓 .md 文档、后续手动收编的文档/代码/数据。
- **真相层** = `~/.loom/data/entries.jsonl`(扁平、人类可读、可 grep;派生的索引/日记都可删可重建)。
- **检索** = SQLite FTS5;**日记** = 按天渲染的 markdown(`vault/journal/`)。
- 采集**只读**、入库前**自动打码**密钥。

## 阶段 C — 收编散落资料
**目标:把不在 git 里的东西也纳入。全部入库前会打码。**

- 文档 / 代码(md/txt/docx/pdf/ipynb/sql/py…,docx/pdf/ipynb 会提取文本成可检索 .md):
  ```
  loom doc add <路径…> --to <类目> [--tags a,b]
  loom doc triage --to inbox           # 让 AI 辅助归类 inbox(输出映射→ --apply 落地)
  ```
- 数据文件(csv/xlsx)→ 蒸馏「数据卡」(列/类型/统计/样例 + 血缘),**原始文件留本地 `_data/` 不上云**:
  ```
  loom data add <文件…> --to <主题> [--code 产出.sql pull.py] [--from 上游.csv] [--used-by 文档]
  ```
  > **为什么分层**:代码/结论是长期资产(上云可搜);原始数据是过程物、可能含敏感明细 → gitignore 留本地。
- 随手散信息(一句话/链接/结论,任何来源统一走这条,**不为单一来源写死采集器**):
  ```
  loom note "<文本>" [--tags a,b]
  ```
- 历史日报(如果用户有导出,如飞书 xlsx)——**仅作一次性种子**,帮你串联历史:
  ```
  loom report import <日报.xlsx>
  ```
  > **注意**:日报**不是持续数据源**。后续日报由 AI 读库合成(见阶段 F),不要把它当采集源。

> ⚠️ `doc add` 目录递归**不扫数据文件**(csv/json/xlsx),避免把几十个数据文件拖进来;数据用 `data add` 显式点名。

## 阶段 D — 云端备份(问用户后再做)
**目标:vault 推私有云 + 每日自动同步。涉及外发,务必先确认。**

1. **问用户**:要不要备份到私有 git 远端?给一个**私有**仓地址(如 GitHub private)。
2. 提醒并确认:
   - vault 里**原始数据 / 二进制 / .env 由代码写的 `.gitignore` 拦住**,不上云;但请用户确认仓是 **private**。
   - **代码仓与数据 vault 是两个仓**(工具可公开,数据永远私有)。
3. 配好后:
   ```
   loom sync --push        # 采集 + 渲染 + 提交 vault + push
   ```
4. 每日 cron(**问用户是否要**,再写):如 `0 19 * * *` 跑 `loom sync --push`。

## 阶段 E — 主题层完整分类(重点:复刻本项目的做法)
**目标:把散落痕迹聚成「一件事」,支持多层 + 多主题。这步决定台账「能不能按主题追溯」。**

### E1. 先和用户定「主题骨架」(域 → 叶子)
条目只打**最具体的叶子**;层级放在**主题页**(`parent:` 可多个 = DAG)。**问用户**在做哪些大块工作,一起定骨架。示例(仅示范结构,按用户真实工作改):
```
数仓建设(伞)  ├ 素材归因 ├ 匹配重构 ├ 成本分摊 ├ BF三方支付(净额/对账) ├ Cohort ├ 数仓底座
分析排查       ├ 反作弊(作弊排查/假单检测) ├ 临时排查
报表周报 · 受众运营 · 工具开发 …
```
> **为什么这样**:"数仓建设"这种大词会横跨一切,所以当**伞**、把具体初期挂它下面;跨两域的
> (如"净额"既属 BF 又被素材引用)用 **DAG 多父**,不用二选一。

### E2. 看候选
```
loom topic gather [--query 关键词] [--since 日期]     # 输出:现有主题 + 待归类条目(带内容片段)
```

### E3. 逐条 AI 判断 + 对抗复核(**核心,别偷懒**)
库里可能上千条,**必须逐条读内容判断**,不能关键词批量。做法:**按 project 分块 → 每块一个 AI 判到骨架叶子 → 再一遍对抗复核剔误挂**(1000+ 条建议用分类工作流并行跑;几十条可顺序手判)。

**必守规则(这些坑本项目都踩过):**
- **日报不塞进单一主题**(它是当天多主题总述,归到主题会污染)。
- **会话是多主题的**:只有当天内容**明显主要是一件事**才给主题,否则不分。
- **别靠单个关键词误挂**("cohort/周报/素材"这类词太泛,会整条卷进来)。
- **拿不准 → 不分**(留空,不硬塞;可写 `none-of-these` 留待人工)。
- **主题页不参与分类**(防自指)。
- **分完必须逐主题读成员核对**:`loom topic show <主题>`,发现误挂就从映射删掉。模糊的会话要**顺 ref 打开原始 jsonl 读那天全部内容**再判——只看摘要/首句会误判。

### E4. 落地 + 建层级 + 核对
```
loom topic apply --file <映射.tsv>     # 每行 `entry_id<TAB>叶子主题1,主题2`;缺失主题自动建页
```
- 给新主题页写 `parent: [[上级]]`(可多父);
- 逐主题核对:
  ```
  loom topic ls                 # 看主题树 + 计数
  loom topic show 数仓建设       # 上卷整棵子树 = "一件事/一个域的全景"
  ```

## 阶段 F — 日常运转
- 每天/定期:`loom sync --push`。
- **AI 写日报**(日报是产出,不是输入):
  ```
  loom report gen <日期>                 # 聚合当天真实痕迹(提交/会话/数据)给你
  # 你(AI)据此写成 今日工作/思考/明日计划 → 存回:
  loom report set <日期> --file <你写的.md>
  ```
- 过时 / 判错的内容:
  ```
  loom deprecate <notes相对路径> [--superseded-by <新版>]   # 默认移入 _attic(移出检索,文件+历史仍在)
  loom deprecate <路径> --mark                              # 仅标记 deprecated,留原处、检索带⚠
  ```
- 想按主题追一件事:`loom topic show <主题>`;想全文搜:`loom search <词> [--project --tool --since]`。

## 一句话回顾
采集(只读+打码)→ 归一到 `entries.jsonl`(唯一真相)→ 派生索引/日记/主题/私有云。
**最花心思的是阶段 E 的分类**:逐条看内容、对抗复核、逐主题核对——这是台账价值的关键,也是最容易翻车的地方。
